# Usability Review And Redesign

Use this when asked to critique, improve, modernize, or redesign an existing UI or frontend.

## Steps

1. Inventory current surface.
   - Capture screenshots, routes, components, states, and target users.
   - Verify: current behavior is understood before proposing changes.
2. Identify task breakdown.
   - Entry, scan, decision, input, confirmation, recovery, and exit.
   - Verify: primary task and most likely failure path are named.
3. Audit heuristics.
   - Use psychology laws, accessibility, responsive, state, content, performance, and ethical checks.
   - Verify: findings map to user impact.
4. Separate severity.
   - Critical: blocks task, trust, accessibility, data loss, safety.
   - Major: causes confusion, delay, repeated error.
   - Minor: polish or consistency.
   - Verify: severity does not reflect personal taste.
5. Propose redesign.
   - Keep working conventions; change only what improves task success or maintainability.
   - Verify: each change has rationale and validation path.
6. Implement or hand off.
   - If coding, use existing components/tokens before inventing new ones.
   - Verify: screenshots and checks cover before/after.

## Review Axes

- Clarity: Can users tell what the screen is for?
- Hierarchy: Is the primary action obvious?
- Familiarity: Does the interaction match expectations?
- Accessibility: Can keyboard, screen-reader, low-vision, and motion-sensitive users complete the task?
- Responsiveness: Does layout survive real content and narrow screens?
- State: Are empty/loading/error/success/disabled handled?
- Trust: Are cost, data use, destructive actions, and AI involvement clear?
- Performance: Does UI respond quickly and avoid layout shifts?
